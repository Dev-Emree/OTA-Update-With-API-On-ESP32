## 2024-05-18 - [CRITICAL] Unencrypted OTA Firmware Update Over HTTP
**Vulnerability:** The firmware updater component (`esp32.ino`) retrieved its binary update payload via a plaintext HTTP connection without any form of cryptographic authentication or signature validation. This exposes the device to a critical Man-in-the-Middle (MITM) attack where an adversary controlling the network can inject malicious firmware, leading to total device compromise.
**Learning:** Embedded systems are often designed with simpler network stacks to save memory, leading developers to neglect basic Transport Layer Security (TLS). However, when the payload being delivered contains executable code (such as OTA updates), it is crucial to establish a secure and authenticated channel. Even a compromised Wi-Fi access point could permanently brick or trojanize the hardware.
**Prevention:** Always use `WiFiClientSecure` alongside an `HTTPS` URL for fetching remote payloads on ESP32 devices. In a production environment, it is also paramount to provide `client.setCACert(rootCACertificate);` rather than calling `client.setInsecure()`, ensuring that the server certificate is cryptographically verified to thwart impersonation.

## 2024-05-20 - [HIGH] Flashing incomplete or HTML payloads during OTA updates
**Vulnerability:** The firmware OTA updater (`esp32.ino`) improperly handled HTTP 301 redirects by treating them as valid firmware binaries, which writes the redirect HTML body to the firmware file. It also lacked validation for the downloaded file size, allowing truncated or corrupted downloads to be flashed, both leading to device bricking (Denial of Service).
**Learning:** Embedded HTTP clients like ESP32's `HTTPClient` do not always automatically follow redirects or natively validate payload integrity. Assuming a non-200 OK response (like 301) or a partially written stream is safe for firmware flashing is a critical error. Moreover, `http.getSize()` returns `-1` for chunked transfer encoding, which must be explicitly handled in validation logic.
**Prevention:** Strictly accept only HTTP 200 OK responses for firmware payloads unless redirect handling is securely configured. Always validate `written == http.getSize()` (handling the `-1` chunked case) and delete the corrupted file using `SD.remove(fileName)` upon failure to prevent flashing bad payloads.
